## Correction — the `streams` row above is void

Both clusters report `input 0 / output 0 / FAIL`, including strimzi. That is a
harness bug, not a result: `topic_record_count` invoked
`kafka-run-class kafka.tools.GetOffsetShell --broker-list`, but `KAFKA_TOOLS`
is a Kafka 4.2 image where that class and flag were removed. Every call
errored, and `2>/dev/null` + `awk '… END{print s+0}'` reported the error as a
confident `0`. Fixed in the skill (`kafka-get-offsets.sh --bootstrap-server`).

Kafka Streams WordCount was validated separately on this same
`0.2.14-preview` deployment (gh #219). With the topology RUNNING and its
internal topics freshly deleted+recreated by `kafka-streams-application-reset`,
200 new input lines drove the whole pipeline:

| topic | before | after |
| --- | ---: | ---: |
| `streams-plaintext-input` | 2524 | 2724 |
| `…-repartition` | 29200 | 38860 |
| `…-changelog` | — | 43663 |
| `streams-wordcount-output` | 85533 | 87333 |

The other scenarios in this report are unaffected — they parse
`kafka-producer-perf-test` / `kafka-consumer-perf-test` output, not offsets.
